## 一、技术领域

本发明属于人工智能代理、社交通信和用户行为分析领域，具体涉及一种数字分身（AI代理）在社交通信中根据用户（分身主人）的实时在线状态和活跃程度，智能决定是否介入回复、何时介入回复的方法。

---

## 二、背景技术

现有AI代理自动回复系统存在以下问题：

（一）**即时回复模式**（传统自动回复、微信"正在忙"回复）：一旦触发条件满足立即回复，无法判断用户是否即将手动回复，导致"抢答"现象——用户正在打字时AI已经替他回了，造成双重回复的尴尬。

（二）**固定延迟模式**（邮件自动回复、某些客服机器人）：设置固定延迟时间后自动回复，但不考虑用户实际状态。用户完全离线时，固定延迟导致不必要的等待；用户在线时，同样的延迟可能仍然导致抢答。

（三）**无状态感知**：现有系统无法区分"用户离线"、"用户在线但在忙别的"、"用户正在跟这个好友聊天"三种截然不同的状态，因此无法做出合理的介入决策。

**空白地带**：没有一个AI代理系统能根据用户的实时在线状态和对话活跃度，动态调整自身的介入时机和行为。

---

## 三、发明内容

### 3.1 技术问题

本发明解决的技术问题是：当数字分身收到好友消息时，如何根据分身主人的实时在线状态和对话活跃度，智能决定是否回复以及何时回复，避免抢答、漏答和不恰当介入。

### 3.2 技术方案

#### 3.2.1 三档状态检测模型

本发明将用户状态分为三档，对应三种不同的分身行为：

| 状态档位 | 检测条件 | 分身行为 |
|----------|----------|----------|
| **Tier 1：离线** | 用户无活跃WebSocket连接 | 立即回复（零延迟） |
| **Tier 2：在线但空闲** | 用户有WebSocket连接，但在设定时间窗口内未向该好友发送过消息 | 延迟N秒后检查，若用户仍未回复则分身介入 |
| **Tier 3：正在对话** | 用户在设定时间窗口内向该好友发送过真人消息 | 分身完全静默，不介入 |

#### 3.2.2 状态检测方法

**Tier 1检测**：通过WebSocket连接管理器查询用户是否有活跃连接：
```
is_online = websocket_manager.is_online(owner_user_id)
```

**Tier 2/3检测**：查询消息数据库，检测用户是否在时间窗口T内向该好友发送过真人消息：
```sql
SELECT COUNT(*) FROM social_messages
WHERE from_user_id = {owner_id}
  AND to_user_id = {friend_id}
  AND sender_mode = 'real'
  AND ai_generated = 0
  AND created_at > datetime('now', '-{T} seconds')
```
- 若 count > 0 → Tier 3（正在对话）
- 若 count = 0 → Tier 2（在线但空闲）

#### 3.2.3 延迟应答机制

当状态为 Tier 2 时，系统执行延迟应答流程：

1. 收到好友消息，检测到用户在线但空闲
2. 创建异步延迟任务，等待N秒（默认30秒）
3. N秒后重新检查：用户是否在这段时间内向该好友发送了真人消息
4. 若用户已回复 → 分身取消介入（用户自己处理了）
5. 若用户仍未回复 → 分身生成回复并发送

```
async function delayed_twin_reply(owner_id, friend_id, message, delay_seconds):
    await sleep(delay_seconds)

    // 重新检查用户是否已回复
    recent_count = query_recent_real_messages(owner_id, friend_id, delay_seconds + buffer)
    if recent_count > 0:
        return  // 用户已回复，分身不介入

    // 用户仍未回复，分身介入
    reply = generate_twin_reply(owner_id, message)
    send_to(friend_id, reply)
    notify_owner(owner_id, friend_id, message, reply)
```

#### 3.2.4 主人通知机制

分身每次自动回复后，通过以下方式通知分身主人：

1. 在分身与主人的私密对话中存储一条通知消息：
```
"刚才{好友名}找你，说：「{消息摘要}」
我替你回了：「{回复摘要}」
具体事情得你来定哦～"
```
2. 通过WebSocket推送 `twin_notification` 事件给主人
3. 更新主人的未读消息计数

#### 3.2.5 行为约束规则

分身自动回复时注入行为约束（social_context），确保分身不越权：
- 不能替主人做决定（定时间、定地点、答应事情）
- 只能传话：「我帮你问问主人」「等主人回来定」
- 语气保持轻松自然

#### 3.2.6 可配置参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `delay_seconds` | Tier 2 延迟等待时间 | 30秒 |
| `active_window` | Tier 3 活跃判断时间窗口 | 35秒 |
| `twin_auto_reply` | 是否启用分身自动回复 | 用户可开关 |

### 3.3 有益效果

（一）首次实现基于用户实时状态的三档自适应AI代理介入策略，避免了"抢答"和"漏答"两种极端情况。

（二）延迟应答机制给予用户合理的反应时间，确保分身只在用户确实无暇时才介入。

（三）主人通知机制使用户回来后能快速了解分身代为处理的所有事项，保持信息透明。

（四）行为约束确保分身不越权，只传话不拍板，维护用户的社交自主权。

## 四、权利要求书

1. 一种基于用户在线状态的数字分身智能延迟应答方法，其特征在于：将用户状态分为"离线"、"在线但空闲"、"正在对话"三档，分别对应"立即回复"、"延迟N秒后条件回复"、"完全静默"三种分身行为策略。

2. 根据权利要求1所述的方法，其特征在于：通过WebSocket连接状态判断用户是否在线，通过查询设定时间窗口内用户是否向特定好友发送过真人消息来区分"在线但空闲"和"正在对话"两种状态。

3. 根据权利要求1所述的方法，其特征在于：延迟应答流程包括创建异步延迟任务、等待设定时间、重新检查用户是否已自行回复、仅在用户未回复时分身才介入回复，确保分身不与用户形成双重回复。

4. 一种数字分身自动回复后的主人通知方法，其特征在于：分身代为回复后，自动在分身与主人的私密对话中存储通知消息（包含好友消息摘要和分身回复摘要），并通过实时通信通道推送通知事件。

5. 根据权利要求1所述的方法，其特征在于：分身自动回复时注入行为约束规则，限制分身不能替主人做决定，只能传话和转达，维护用户的社交自主权。

## 五、在先技术对比

| 在先技术 | 与本发明的区别 |
|----------|---------------|
| 微信自动回复 | 固定模板回复，无状态感知，无延迟检查机制 |
| 企业微信/钉钉机器人 | 即时触发，不考虑用户是否正在回复 |
| Replika / Character.ai | AI伴侣，非代理模式，不需判断用户状态 |
| 邮件自动回复 (OOF) | 固定延迟，不区分在线/离线/活跃状态 |
| Google Smart Reply | 建议式回复，不自动发送，非代理模式 |

本发明的独创性：首次实现基于用户三档在线状态的自适应AI代理延迟介入策略，结合延迟等待-重新检查机制避免双重回复，并通过主人通知系统保持信息透明。
