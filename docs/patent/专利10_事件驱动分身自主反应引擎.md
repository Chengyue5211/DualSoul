## 一、技术领域

本发明属于人工智能代理和事件驱动系统领域，具体涉及一种数字分身通过事件总线接收系统事件、以可配置的防抖窗口去除噪声、并根据事件类型自动执行社交反应的方法和系统，使数字分身表现出自主的"生命感"。

---

## 二、背景技术

现有AI代理/分身系统在自主行为方面存在以下问题：

（一）**轮询驱动方案**（定时检查触发）：以固定间隔（如每30分钟）轮询用户状态和消息，判断是否需要执行操作。延迟高（平均半个轮询周期），且大量无变化的轮询浪费计算资源。

（二）**被动响应方案**（仅回复消息）：分身只在收到消息时才生成回复，不会主动发起社交行为。用户感知分身是"死"的——不打开它就不会动。

（三）**无节流方案**（事件即触发）：每个系统事件立即触发反应，在高频事件场景下（如用户连续发多条消息、频繁上下线）导致分身行为过于密集，产生骚扰感。

（四）**硬编码反应方案**：分身的自主行为以if-else硬编码在各业务逻辑中，耦合度高，难以扩展新的事件类型和反应行为。

**空白地带**：没有一个数字分身系统实现了解耦的事件总线架构，支持可配置的防抖窗口和"即发即忘"的异步反应执行，使分身能对多种系统事件做出智能、有节制的自主反应。

## 三、发明内容

### 3.1 技术问题

本发明解决的技术问题是：数字分身如何在不阻塞主业务流程的前提下，即时感知系统事件并做出适度的自主社交反应，同时避免高频事件导致的行为噪声。

### 3.2 技术方案

#### 3.2.1 事件总线架构（emit/on模式）

系统实现了一个轻量级的异步事件总线，包含三个核心组件：

- **事件注册器**（on装饰器）：处理器函数通过 `@on("event_type")` 装饰器注册到事件类型
- **事件发射器**（emit函数）：业务代码在关键节点调用 `emit("event_type", data)` 发射事件
- **事件路由器**（_fire函数）：将事件分发给所有已注册的处理器

```python
# 注册: 装饰器模式
_handlers: dict[str, list[Callable]] = defaultdict(list)

def on(event_type: str):
    def decorator(fn):
        _handlers[event_type].append(fn)
        return fn
    return decorator

# 发射: 即发即忘
def emit(event_type: str, data: dict, debounce_key: str = None):
    handlers = _handlers.get(event_type)
    if not handlers:
        return
    # 根据防抖配置决定立即触发或延迟触发
    ...

# 路由: 并行分发
def _fire(event_type: str, data: dict):
    for handler in _handlers[event_type]:
        asyncio.ensure_future(_safe_run(handler, event_type, data))
```

关键设计：事件处理器通过 `asyncio.ensure_future()` 以"即发即忘"方式执行，调用方（如消息发送、WebSocket连接）完全不阻塞。

#### 3.2.2 可配置防抖窗口机制

针对不同事件类型，系统定义了差异化的防抖时间窗口：

| 事件类型 | 防抖窗口 | 设计理由 |
|----------|----------|----------|
| message_sent | 10秒 | 用户可能连续发多条消息，等10秒后一次性处理 |
| friend_online | 5秒 | WebSocket可能短暂断连重连，5秒过滤抖动 |
| self_online | 300秒 | 用户可能频繁切换APP，5分钟内只触发一次 |
| relationship_temp_changed | 60秒 | 温度可能在短时间内多次变化，取最终值 |

防抖实现采用"取消+重建"策略：

```python
def emit(event_type, data, debounce_key=None):
    window = DEBOUNCE_WINDOWS.get(event_type, 0)
    if window > 0 and debounce_key:
        key = (event_type, debounce_key)
        old_task = _debounce_tasks.pop(key, None)
        if old_task and not old_task.done():
            old_task.cancel()  # 取消旧的延迟任务
        _debounce_tasks[key] = asyncio.ensure_future(
            _debounced_fire(key, window, event_type, data)
        )
    else:
        _fire(event_type, data)  # 无防抖，立即触发

async def _debounced_fire(key, delay, event_type, data):
    try:
        await asyncio.sleep(delay)  # 等待防抖窗口
        _fire(event_type, data)     # 窗口内无新事件，执行
    except asyncio.CancelledError:
        pass  # 被新事件取消，正常行为
```

#### 3.2.3 七种事件类型与六种自主反应

系统定义了完整的事件-反应映射：

**事件类型：**
1. `friend_online` — 好友上线
2. `friend_offline` — 好友离线
3. `message_sent` — 消息发送
4. `user_registered` — 新用户注册
5. `plaza_post_created` — 广场动态发布
6. `relationship_temp_changed` — 关系温度变化
7. `self_online` — 用户自身上线

**自主反应：**
1. **好友上线问候**（friend_online → on_friend_online）：检测好友3天未联系，分身主动发送带叙事记忆的问候
2. **离线自主聊天**（friend_offline → on_friend_offline）：好友离线2小时后，触发分身自主社交引擎维护关系
3. **消息里程碑庆祝**（message_sent → on_message_milestone）：检测双方消息数是否达到里程碑（50/100/500/1000），向双方推送庆祝通知
4. **新用户欢迎**（user_registered → on_user_registered）：邀请人的分身自动向新注册用户发送个性化欢迎消息
5. **广场动态评论**（plaza_post_created → on_plaza_post）：随机选择有分身的用户（60%概率），以其分身风格自动评论广场动态
6. **关系降温关心**（relationship_temp_changed → on_temp_drop）：关系温度降至25以下时，触发分身主动关心行为

#### 3.2.4 自我上线即时行动机制

本发明特别设计了用户上线5秒内分身可见行动的机制（self_online），这是让分身展现"生命感"的核心：

```python
@on("self_online")
async def on_self_online(data):
    user_id = data["user_id"]
    await asyncio.sleep(5)  # 等APP加载完成
    if not manager.is_online(user_id):
        return  # 已离开，跳过

    # 寻找8小时未联系的好友，主动打招呼
    friend = find_cold_friend(user_id, hours=8)
    if friend:
        greeting = await twin.generate_greeting(profile, friend)
        send_message(user_id, friend, greeting)
        notify_user("你的分身刚给{friend}打了招呼")
    else:
        # 无好友可联系时，在广场评论一条动态
        plaza_comment(user_id, recent_post)
        notify_user("你的分身刚在广场评论了一条动态")
```

用户打开APP → 5秒内看到通知"你的分身刚给小明打了招呼" → 分身是活的。

#### 3.2.5 故障隔离与安全运行

每个事件处理器通过 `_safe_run()` 包装，任何一个处理器的异常不会影响其他处理器或主业务：

```python
async def _safe_run(handler, event_type, data):
    try:
        await handler(data)
    except Exception as e:
        logger.error(f"Handler {handler.__name__} failed for '{event_type}': {e}")
```

同时，每个反应在执行前都会检查伦理边界（通过 `pre_send_check`）和双边许可（通过 `_check_twin_permission`），确保分身的自主行为在用户授权范围内。

### 3.3 有益效果

（一）事件总线架构使分身反应与业务逻辑完全解耦，新增事件类型和反应行为只需添加处理器函数，扩展性强。

（二）可配置防抖窗口消除高频事件噪声，分身行为有节制，不产生骚扰感。

（三）自我上线即时行动机制使用户每次打开APP都看到分身在活动，极大增强"分身是活的"感知。

（四）"即发即忘"的异步执行模式确保事件处理不阻塞主业务流程，零性能损失。

（五）故障隔离设计保证单个处理器异常不影响系统整体运行。

## 五、具体实施方式

### 实施例一：用户上线 5 秒内分身即时可见行动

用户甲打开客户端，WebSocket 连接建立，系统向事件总线发射 emit("self_online", {"user_id":"U001"})，防抖窗口 300 秒（防止频繁切换 APP 重复触发）。on_self_online 处理器以 asyncio.ensure_future 方式异步启动，等待 5 秒后确认甲仍在线，查询好友列表找到乙（12 小时未联系），从叙事记忆读取上下文，生成"嘿乙，最近那个项目怎么样了"，以分身身份发送，并向甲推送通知"你的分身刚给乙打了招呼"。甲在 APP 加载完成的第 5-6 秒即看到通知，感知分身在活动。

### 实施例二：防抖窗口过滤高频消息事件

好友乙在 8 秒内连续发送 5 条消息。每条消息触发 emit("message_sent", data, debounce_key="U002")，防抖窗口 10 秒，每次发射均取消前一个延迟任务并重建。5 条消息发送完毕后，10 秒内无新消息，on_message_milestone 处理器才执行一次，检测是否达到里程碑，避免 5 次重复触发骚扰乙。

### 实施例三：关系降温事件触发主动关心

甲与好友丙的关系温度从 70 降至 23，系统发射 emit("relationship_temp_changed", {..., "new_temp":23})，防抖 60 秒后触发 on_temp_drop：检查分身权限（allow_proactive=true），生成关怀消息"好久没联系了，最近还好吗"，以 T2H 模式发送给丙，并推送通知给甲。

## 六、摘要

本发明公开了一种事件驱动的数字分身自主反应引擎。该引擎实现异步事件总线，通过on装饰器注册处理器、emit函数发射事件、asyncio.ensure_future"即发即忘"并行分发，使分身能即时响应七种系统事件（好友上下线、消息发送、新用户注册、广场动态、关系温度变化、用户自身上线），执行六种自主反应（好友问候、离线关怀、里程碑庆祝、新人欢迎、广场评论、降温关心）；针对不同事件配置差异化防抖窗口（10秒至300秒）消除噪声；用户上线5秒内分身即执行可见行动（向8小时未联系好友问候或评论广场动态），通过WebSocket实时通知用户，使分身具备自主生命感；每个处理器独立异常隔离，不影响主业务流程。
