## 一、技术领域

本发明属于人工智能代理和对话系统领域，具体涉及一种数字分身在代替用户进行社交回复时，自动检索并利用历史对话上下文，生成连贯、有记忆的回复的方法和系统。

---

## 二、背景技术

现有AI代理/分身自动回复系统存在以下问题：

（一）**无记忆回复**（传统自动回复、简单机器人）：每次收到消息都作为独立事件处理，AI不知道之前聊了什么，回复可能与对话上下文矛盾或重复。

（二）**全量上下文方案**（某些聊天机器人）：将所有历史消息作为上下文输入AI，但在长期社交场景中消息量巨大，导致成本高、延迟大、且可能包含过多无关信息。

（三）**孤立prompt方案**：仅用当前消息和用户性格画像构建提示词，AI回复虽符合性格但缺乏对话连贯性。当好友连续发多条消息时，分身可能对每条消息重复类似回复，或给出与之前回复矛盾的内容。

**空白地带**：没有一个数字分身系统能在保持低延迟、低成本的前提下，智能地检索和注入适量的对话上下文，使分身回复既符合主人性格又与对话流程连贯。

## 三、发明内容

### 3.1 技术问题

本发明解决的技术问题是：数字分身在代替用户回复时，如何高效获取并利用相关对话历史，使回复具有上下文连贯性，避免重复、矛盾或脱节。

### 3.2 技术方案

#### 3.2.1 滑动窗口对话检索

本发明采用滑动窗口机制，从消息数据库中检索分身主人与当前对话好友之间的最近N条消息作为上下文：

```sql
SELECT from_user_id, content, sender_mode
FROM social_messages
WHERE ((from_user_id={owner} AND to_user_id={friend})
    OR (from_user_id={friend} AND to_user_id={owner}))
  AND msg_type='text' AND content != ''
ORDER BY created_at DESC
LIMIT {N}
```

其中N为可配置的窗口大小（默认值6条），平衡上下文丰富度与AI调用成本。

#### 3.2.2 角色映射与消息构建

检索到的历史消息按时间正序排列，并进行角色映射：
- 分身主人发送的消息（无论真人还是分身模式）→ 映射为 `assistant` 角色
- 好友发送的消息 → 映射为 `user` 角色

构建多轮对话消息数组：
```json
[
  {"role": "system", "content": "人格画像+行为规则"},
  {"role": "user", "content": "好友第1条消息"},
  {"role": "assistant", "content": "主人/分身的回复"},
  {"role": "user", "content": "好友第2条消息"},
  {"role": "assistant", "content": "主人/分身的回复"},
  {"role": "user", "content": "好友最新消息（当前待回复）"}
]
```

#### 3.2.3 去重机制

当前待回复消息可能已经存在于检索到的历史记录中（因为消息先存库再触发回复）。系统自动检测并去除重复：
```
if history[-1].content == incoming_message:
    history = history[:-1]  // 移除末尾重复
```

#### 3.2.4 系统提示词中的防重复指令

在系统提示词中明确注入防重复规则：
```
"Only respond to the LATEST message, do not recap previous messages."
```
防止AI模型在看到多条历史消息后，试图对每条消息都进行回应，导致回复冗长混乱。

#### 3.2.5 双重上下文融合

分身回复时同时融合两种上下文：
1. **人格上下文**：用户的性格画像、说话风格（静态）
2. **对话上下文**：最近的聊天记录（动态）

两者在AI提示词中的位置分工：
- 人格上下文 → system prompt（设定分身身份和行为基线）
- 对话上下文 → message history（提供对话流信息）
- 当前消息 → 最后一条user message（触发生成回复）

#### 3.2.6 分身自聊场景的扩展上下文

当用户与自己的分身对话时（分身自聊），系统注入更丰富的上下文：
- 最近8轮对话历史（比好友对话的6条更多）
- 好友列表上下文（分身知道主人有哪些好友）
- 冷联系好友提醒（随机概率提醒主人联系久未互动的好友）

### 3.3 有益效果

（一）分身回复具有对话连贯性，不会重复或矛盾，社交体验自然流畅。

（二）滑动窗口机制控制上下文大小，平衡回复质量与API调用成本。

（三）去重机制和防重复指令双重保障，避免AI对历史消息进行不必要的复述。

（四）角色映射使AI能准确理解对话中"谁说了什么"，生成符合对话流程的回复。

## 四、权利要求书

1. 一种具备对话记忆的数字分身应答方法，其特征在于：在数字分身生成自动回复时，自动检索分身主人与当前对话好友之间的最近N条消息作为上下文，将历史消息按角色映射（主人→assistant，好友→user）构建多轮对话消息数组，使AI模型在生成回复时能感知对话流程和上下文。

2. 根据权利要求1所述的方法，其特征在于：采用滑动窗口机制控制上下文大小（默认6条），检索结果按时间正序排列，并自动检测去除与当前待回复消息重复的末尾记录。

3. 根据权利要求1所述的方法，其特征在于：在系统提示词中同时融合静态人格上下文（性格画像、说话风格）和动态对话上下文（近期聊天记录），两者分别通过system prompt和message history注入AI模型。

4. 根据权利要求1所述的方法，其特征在于：在系统提示词中注入防重复指令，限制AI模型只回应最新消息而不复述历史消息，防止回复冗长混乱。

5. 一种数字分身自聊场景的扩展上下文方法，其特征在于：当用户与自身分身对话时，注入更大的对话窗口、好友列表上下文和冷联系好友提醒，使分身具备主动社交维护能力。

## 五、在先技术对比

| 在先技术 | 与本发明的区别 |
|----------|---------------|
| ChatGPT / Claude 对话 | 用户自主对话，非代理模式，不涉及替用户回复好友 |
| RAG（检索增强生成） | 检索知识库文档，不是检索社交对话历史 |
| 微软Copilot邮件建议 | 基于邮件线程上下文，但不涉及实时社交和人格保真 |
| 企业微信智能回复 | 基于FAQ匹配，无多轮对话上下文检索 |
| Replika对话 | AI伴侣自身记忆，不是代替用户与第三方社交 |

本发明的独创性：首次在数字分身代理社交场景中实现了基于滑动窗口的对话历史检索和角色映射机制，使分身在代替用户回复好友时具备对话记忆和上下文连贯性。
