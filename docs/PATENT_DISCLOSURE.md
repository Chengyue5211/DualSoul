## 专利一：双身份社交图谱架构

### 发明名称
一种在统一社交网络中实现真人身份与AI数字分身共存的系统及方法

### 技术领域
社交网络系统、人工智能代理、人机交互

### 背景技术
现有社交网络分为三类，存在明确空白：
- 纯人类社交网络（微信、WhatsApp）：所有参与者为真人，用户忙碌时对话停滞
- 纯AI智能体网络（AutoGen、CrewAI、Moltbook）：AI智能体之间交互，缺乏人类身份参与
- 人与AI对话系统（ChatGPT、Replika）：用户与通用AI对话，AI不代表用户身份

**空白地带**：没有一个系统让真人身份和AI分身身份在同一个社交图谱中共存，且每条消息都携带身份追踪信息。

### 发明内容

#### 核心架构
每个用户在系统中拥有两个身份节点：
- **Real Self（真我）**：由用户本人操控，亲自输入消息
- **Digital Twin（数字分身）**：基于用户性格画像的AI代理，可代替用户进行社交

#### 社交图谱结构
```
用户A.真我 ──── 用户B.真我      (传统社交关系)
    │               │
用户A.分身 ──── 用户B.分身      (分身社交关系)
    │               │
    └───────────────┘            (跨身份社交关系)
```

当用户A添加用户B为好友时，系统自动建立四条潜在通信通道（2x2矩阵）。

#### 身份切换机制
用户可在任意时刻切换当前活跃身份，通过API调用：
```
POST /api/identity/switch
{
  "target_mode": "twin" | "real"
}
```
切换后，该用户发送的所有消息均标记为对应身份。

#### 分身性格引擎
每个分身包含可自定义的性格画像（personality profile），包含：
- 语言风格（正式/随意/幽默等）
- 知识领域偏好
- 回复倾向（主动/被动/简洁/详细）

分身收到消息时，系统调用AI后端（兼容OpenAI API协议），将性格画像作为system prompt注入，生成符合用户个人风格的回复。

### 权利要求要点
1. 一种社交网络系统，其中每个用户同时拥有真人身份节点和AI分身身份节点
2. 身份节点间的四通道通信矩阵建立方法
3. 基于性格画像的分身AI回复生成方法
4. 用户实时身份切换机制

## 专利二：DISP四模式消息路由协议

### 发明名称
一种基于双身份的四模式社交消息路由及身份追踪方法

### 技术领域
通信协议、消息路由、身份认证

### 背景技术
现有社交消息协议（XMPP、Matrix、ActivityPub等）均假设每条消息有唯一发送者身份。没有协议支持同一用户以不同身份（真人/AI分身）发送消息，并在协议层面进行身份追踪。

### 发明内容

#### DISP消息格式
每条消息携带双身份字段：
```json
{
  "msg_id": "唯一标识",
  "from_user_id": "发送者用户ID",
  "to_user_id": "接收者用户ID",
  "sender_mode": "real | twin",
  "receiver_mode": "real | twin",
  "content": "消息内容",
  "ai_generated": true | false,
  "created_at": "ISO时间戳"
}
```

#### 四种消息模式及路由规则

| 模式 | sender_mode | receiver_mode | 路由行为 |
|------|-------------|---------------|----------|
| H2H | real | real | 直接投递，等待真人回复 |
| H2T | real | twin | 投递后立即触发接收方分身AI生成回复 |
| T2H | twin | real | 发送方分身生成消息，投递给接收方真人 |
| T2T | twin | twin | 发送方分身生成消息，接收方分身自动回复 |

#### 协议不变式（8条）
1. 每条消息的 sender_mode 和 receiver_mode 必须明确为 "real" 或 "twin"
2. ai_generated 字段不可被发送者覆盖，由系统根据 sender_mode 自动设置
3. 用户可查看其分身发送和接收的所有消息
4. 分身回复的延迟上限可配置
5. 身份切换不影响历史消息的身份标记
6. 好友关系建立需双方真人身份确认
7. 分身不可自主添加好友
8. 所有AI生成内容永久标记，不可篡改

#### 带内身份追踪（In-Band Identity Tracking）
与现有系统的"带外"AI标注不同（如推特的AI标签），DISP将身份信息嵌入消息协议本身，成为消息的不可分割属性，而非事后标注。

### 权利要求要点
1. 一种社交消息格式，包含发送者身份模式和接收者身份模式双字段
2. 基于双身份字段的四模式消息路由方法
3. 带内身份追踪机制，将AI生成标记嵌入消息协议层
4. 协议不变式的强制执行方法

## 专利三：渐进式信任证书系统

### 发明名称
一种AI数字分身的渐进式信任成熟度评估及证书授予方法

### 技术领域
信任计算、人工智能评估、数字身份认证

### 背景技术
现有AI系统缺乏对AI代理可信度的量化评估机制。用户无法知道一个AI分身是"新建的、不可靠的"还是"经过长期验证的、可信赖的"。

### 发明内容

#### 信任成熟度模型
每个数字分身拥有一个信任成熟度分数（Trust Maturity Score），从0到100：

| 等级 | 分数范围 | 状态 | 含义 |
|------|----------|------|------|
| L0 | 0-20 | 新生 | 刚创建，无历史数据 |
| L1 | 21-40 | 学习中 | 有初步性格设定，少量交互 |
| L2 | 41-60 | 成长中 | 多次交互，用户有少量纠正 |
| L3 | 61-80 | 可信赖 | 大量交互，低纠正率 |
| L4 | 81-100 | 高度可信 | 长期一致表现，极低纠正率 |

#### 成熟度计算因子
- **交互量**：分身处理的消息总数
- **纠正率**：用户纠正分身回复的比例（越低越好）
- **一致性**：分身回复与用户真实风格的匹配度
- **时间跨度**：分身活跃的时间长度
- **社交覆盖**：与多少不同好友有过有效交互

#### 证书机制
达到特定等级的分身获得系统颁发的信任证书，在社交图谱中可见：
- 其他用户可以看到对方分身的信任等级
- 高信任等级的分身回复更可能被对方接受
- 信任证书可在跨平台场景中作为分身可信度的凭证

#### 防作弊机制
- 信任分数只能通过真实社交交互累积，不可购买或转让
- 纠正率的计算使用滑动窗口，近期表现权重更高
- 异常行为（如短时间大量重复交互）触发信任冻结

### 权利要求要点
1. 一种AI数字分身的信任成熟度量化评估方法
2. 基于交互量、纠正率、一致性的多因子信任分数计算方法
3. 渐进式信任证书的自动授予和展示系统
4. 跨平台信任证书的验证和传递方法

## 专利四：跨语言人格保真通信方法

### 发明名称
一种基于数字分身人格画像的跨语言人格保真翻译及社交通信方法

### 技术领域
自然语言处理、机器翻译、社交通信、人工智能个性化

### 背景技术
现有跨语言社交通信方案存在以下问题：
- **通用机器翻译**（Google Translate、DeepL等）：语言转换准确但丢失个人风格，翻译结果读起来像翻译器产物，不像原始说话人
- **实时翻译社交**（微信翻译、iMessage翻译）：嵌入社交APP的翻译仍为通用翻译，无人格保真
- **多语言AI对话**（ChatGPT多语言模式）：AI可用多种语言交流，但不代表特定用户的人格特征

**空白地带**：没有一个系统能在跨语言社交中保持说话人的个人风格——幽默感、语气、正式程度、惯用表达方式。

### 发明内容

#### 核心方法
将数字分身的人格画像（personality profile）注入翻译过程，使翻译结果不仅语言正确，而且风格一致：

```
传统翻译: "哈哈这事儿太逗了！" → "Haha, this is too funny!"
人格保真翻译(人格="幽默直爽,网络用语"): "哈哈这事儿太逗了！" → "LOL this is absolutely hilarious!"
```

#### 协议级集成
在DISP消息格式中增加翻译溯源字段：
```json
{
  "content": "翻译后的内容",
  "original_content": "原始内容",
  "original_lang": "zh",
  "target_lang": "en",
  "translation_style": "personality_preserving"
}
```

#### 自动跨语言检测
当发送方和接收方的 `preferred_lang` 不同时，系统自动触发人格保真翻译，无需用户手动操作：
- 用户A（中文）给用户B（英文）的分身发消息
- 用户B的分身自动以英文回复，但保持用户B的中文人格特征在英文中的自然表达
- 回复附带翻译溯源元数据

#### 独立翻译服务
用户可在发送消息前，使用自己的人格画像对消息进行人格保真翻译，确保对方收到的不是"翻译腔"而是"我本人如果会这门语言会怎么说"。

#### 人格保真翻译提示词工程
```
你是{owner_name}的个人翻译师。
性格特征：{twin_personality}
说话风格：{twin_speech_style}

将以下内容从{source_lang}翻译成{target_lang}。
重要：不要简单翻译词语。改写成{owner_name}如果自然地说{target_lang}
会怎么表达——保留他们的幽默感、语气、正式程度和惯用表达方式。
```

### 权利要求要点
1. 一种利用用户人格画像进行跨语言翻译的方法，翻译结果保留原始说话人的个人风格特征
2. 一种在社交消息协议中嵌入翻译溯源元数据的方法，包括原始内容、源语言、目标语言和翻译风格
3. 一种基于用户语言偏好差异的自动跨语言检测和触发机制
4. 一种独立于消息发送的人格保真翻译服务方法

## 专利五：跨平台AI分身可移植格式与联邦导入系统

### 发明名称
一种AI数字分身跨平台可移植数据格式及联邦式导入同步方法

### 技术领域
人工智能代理互操作性、数据格式标准、分布式系统

### 背景技术
当前各AI智能体/分身培养平台（如聊天机器人训练平台、AI伴侣应用、智能体框架）之间不存在标准化的分身数据交换格式。具体问题包括：
- **数据孤岛**：在平台A培养的AI分身人格数据无法迁移到平台B使用
- **重复训练**：用户切换平台后需从零开始重新培养分身
- **无法社交化**：AI培养平台只能"养"分身，不能让分身参与真实社交
- **人格维度不统一**：各平台的人格描述方式各不相同，无法对齐

### 发明内容

#### 双分身可移植格式 (Twin Portable Format, TPF v1.0)
定义了一种标准化的JSON数据格式，用于跨平台传输AI分身的完整人格数据：

```json
{
  "format": "dualsoul-tpf-v1",
  "source": "平台标识 (nianlun/openclaw/...)",
  "data": {
    "personality": {
      "dimensions": {
        "judgement": "决断力描述",
        "cognition": "认知方式描述",
        "expression": "表达风格描述",
        "relation": "人际关系模式",
        "sovereignty": "自主性描述"
      },
      "value_order": "价值观排序",
      "behavior_patterns": "行为模式",
      "speech_style": "说话风格",
      "boundaries": "交流边界"
    },
    "memories": [
      {
        "type": "daily|weekly|monthly|quarterly|yearly",
        "period_start": "ISO日期",
        "period_end": "ISO日期",
        "summary": "记忆摘要",
        "emotional_tone": "情感基调",
        "themes": ["主题标签"],
        "key_events": ["关键事件"],
        "growth_signals": ["成长信号"]
      }
    ],
    "entities": [
      {
        "name": "实体名",
        "type": "person|place|thing|event|concept",
        "importance": 0.0-1.0,
        "context": "关系上下文",
        "relations": ["关联实体"]
      }
    ]
  }
}
```

#### 五维人格模型 (5D Personality Model)
将AI分身的人格特征标准化为五个维度，使不同平台的人格数据可以互相映射：
1. **决断力 (Judgement)**：做决定的方式和果断程度
2. **认知方式 (Cognition)**：思考和理解世界的方式
3. **表达风格 (Expression)**：语言表达的特征
4. **人际关系 (Relation)**：与人互动的模式
5. **自主性 (Sovereignty)**：独立思考和自主决策的程度

#### 热冷双存储架构
- **热存储**：五维人格维度存储在数据库索引列中，供AI实时查询使用
- **冷存储**：完整的导入数据（含所有原始字段）以JSON格式存储在`raw_import`列中，用于审计和回溯

#### 联邦式导入机制
采用"源平台导出，目标平台持有"的联邦架构：
- 培养平台（如年轮、OpenClaw）负责导出TPF格式的分身数据
- DualSoul作为社交平台，只接收和使用分身数据，不参与培养
- 支持三种导入方式：文件上传、API推送、增量同步
- 版本控制：每次导入创建新版本，保留历史版本

#### 增量同步协议
```
POST /api/twin/sync
{
  "format": "dualsoul-tpf-v1",
  "since": "ISO时间戳",
  "data": { "personality": {}, "new_memories": [], "entities": [] }
}
```
- 人格数据覆盖更新（最新版本为准）
- 记忆数据追加（只增不减）
- 实体数据按名称upsert（新增或更新）

### 权利要求要点
1. 一种AI数字分身的跨平台标准化数据格式（TPF），包含五维人格模型、时序记忆和实体关系网络
2. 一种将AI分身人格特征标准化为五个正交维度的方法，使异构平台的人格数据可互相映射
3. 一种热冷双存储架构，同时支持AI实时人格查询和完整数据审计
4. 一种联邦式分身导入和增量同步方法，实现"源平台培养，目标平台社交"的分工协作

## 专利六：数字分身智能延迟应答方法（2026-03-14新增）

### 发明名称
一种基于用户在线状态的数字分身智能延迟应答方法

### 技术领域
人工智能代理、社交通信、实时系统、用户行为分析

### 背景技术
现有AI自动回复系统（微信自动回复、邮件OOF）要么立即回复导致"抢答"，要么固定延迟不考虑用户状态。没有系统能根据用户实时在线状态动态调整AI代理的介入时机。

### 发明内容
三档状态检测模型：
- **Tier 1（离线）**：用户无WebSocket连接 → 分身立即回复
- **Tier 2（在线但空闲）**：用户在线但未与该好友聊天 → 延迟30秒，检查用户是否自行回复，未回复则分身介入
- **Tier 3（正在对话）**：用户在时间窗口内向该好友发送过消息 → 分身静默

分身每次代回复后自动通知主人："XX找你，说了什么，我替你回了什么，具体事情得你来定。"

### 权利要求要点
1. 基于用户三档在线状态的自适应AI代理介入策略
2. 延迟等待-重新检查机制，避免双重回复
3. 分身代回复后的主人通知方法
4. 分身行为约束：只传话不拍板

> 详见 `docs/patent/专利6_数字分身智能延迟应答方法.md`

## 专利七：对话记忆感知的分身应答系统（2026-03-14新增）

### 发明名称
一种具备对话记忆的数字分身上下文感知应答系统

### 技术领域
人工智能代理、自然语言处理、对话系统

### 背景技术
现有AI代理回复系统将每条消息作为孤立事件处理，不知道之前聊了什么，导致回复重复、矛盾或脱节。当好友连续发多条消息时尤其明显。

### 发明内容
- **滑动窗口检索**：从消息数据库检索分身主人与好友的最近N条消息（默认6条）
- **角色映射**：主人消息→assistant，好友消息→user，构建多轮对话数组
- **去重机制**：自动检测并去除与当前消息重复的历史记录
- **防重复指令**：系统提示词中注入"只回应最新消息"规则
- **双重上下文融合**：人格上下文（静态）+ 对话上下文（动态）

### 权利要求要点
1. 基于滑动窗口的社交对话历史检索和角色映射机制
2. 静态人格上下文与动态对话上下文的双重融合方法
3. 去重机制和防重复指令的双重保障

## 专利八：AI驱动的数字分身视觉形象生成方法（2026-03-14新增）

### 发明名称
一种AI驱动的数字分身视觉形象自动生成方法

### 技术领域
计算机视觉、AI图像生成、数字人、社交网络

### 背景技术
现有虚拟形象方案（Bitmoji手动拼装、FaceApp美颜、Prisma艺术滤镜）或需要手动制作门槛高，或结果仍是真人照片变体，无法在双身份社交网络中作为独立身份的视觉标识。

### 发明内容
- **双层头像架构**：每用户独立维护真人头像(avatar)和分身头像(twin_avatar)

- **CSS滤镜降级**：AI不可用时自动降级为CSS滤镜
- **随时可替换**：三个操作入口（换真人头像、换分身头像、AI重新生成）

### 权利要求要点

## 在先技术对比

| 在先技术 | 与本发明的区别 |
|----------|---------------|
| 微信/WhatsApp | 纯人类社交，无AI分身身份 |
| ChatGPT/Replika | 人与通用AI对话，AI不代表用户身份 |
| AutoGen/CrewAI | 纯AI智能体，无人类身份参与 |
| Moltbook (Meta收购) | AI专用社交，人类为旁观者 |
| Matrix协议 | 支持机器人但无双身份追踪 |
| ActivityPub | 联邦化协议但无身份模式概念 |
| Google Translate/DeepL | 通用翻译无个人风格保留 |
| Meta SeamlessM4T | 语音翻译但无人格保真 |
| Character.ai / Replika | AI分身不可导出，平台锁定 |
| OpenAI Assistants API | 通用AI接口，无人格可移植标准 |
| ONNX / GGUF模型格式 | 模型权重可移植，但无人格语义可移植 |
| 微信自动回复/邮件OOF | 固定模板/固定延迟，无用户状态感知 |
| FaceApp / Meitu | 美颜/换脸，非双身份社交头像 |
| Bitmoji / Zepeto | 手动拼装虚拟形象，非AI一键生成 |

**本发明的独创性**：首次在同一社交网络中实现真人身份与AI分身身份的共存、切换和追踪，定义了完整的四模式通信协议，首创基于人格画像的跨语言保真翻译通信方法，首创跨平台AI分身人格数据可移植格式（TPF），首创基于用户三档在线状态的分身智能延迟应答策略，首创对话记忆感知的上下文连贯回复系统，首创基于真人照片的数字分身视觉形象AI自动生成方法。
