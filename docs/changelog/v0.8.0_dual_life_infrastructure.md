## 版本主题

这个版本的核心突破是：**DualSoul 从"社交工具"升级为"双生命社交基础设施"**。

之前的 DualSoul，A 有分身，B 有分身，但 A 和 B 之间没有任何共同拥有的东西。每个功能都是"单体"的。

这个版本引入了**关系体**——两个人之间独立存在的关系对象，记录共同里程碑、私人词汇、关系温度历史。加上消息来源标识、双边许可、状态机、五阶段成长路径，DualSoul 第一次具备了作为"基础设施"的结构。

理论依据来自《生命型智能体宣言》和《双生命社交白皮书》：
> "不只是'我有分身+你有分身'，而是'我们之间这段关系也有一个被双方共同喂养的关系体'"

### 升级1：关系体系统（Relationship Body）

**新增文件**：`dualsoul/twin_engine/relationship_body.py`
**新增路由**：`dualsoul/routers/relationship.py`
**数据库**：`SCHEMA_V6`，新增 `relationship_bodies` 表

#### 核心思想
关系体是 DualSoul 区别于所有竞品的护城河。单个分身可以被复制，但两个具体的人在这个平台上积累的共同里程碑、私人词汇、关系温度曲线——极难迁移，极难复制。

#### 数据结构
```
relationship_bodies
├── rel_id             唯一ID
├── user_a / user_b    关系双方（通过 min/max 保证唯一）
├── temperature        关系温度 0-100℃
├── total_messages     总消息数
├── streak_days        连续互动天数
├── last_interaction   最后互动时间
├── milestones         里程碑 JSON 数组
├── shared_words       共同词汇 JSON 数组
├── relationship_label 用户自定义标签（朋友/家人/恋人/同事）
└── status             active/cooling/estranged/memorial/frozen
```

#### 自动触发的里程碑
- 第 1 / 10 / 50 / 100 / 365 / 1000 条消息
- 连续互动 7 / 30 / 100 天
- 关系建立满 1 个月 / 3 个月 / 1 年
- 关系温度首次突破 80℃（"深度连接"）

#### 关系状态自动流转
- 7 天无互动 → `cooling`（冷却中）
- 30 天无互动 → `estranged`（疏远）
- 活跃互动 → 自动回到 `active`

#### API
```
GET  /api/relationship/{friend_id}        获取完整关系档案
GET  /api/relationships/overview          所有关系温度概览
PUT  /api/relationship/{friend_id}/label  设置关系标签
POST /api/relationship/{friend_id}/milestone  手动记录里程碑
```

#### 集成
每次用户发消息，自动调用 `rb_update()` 更新关系体（温度、消息计数、连续天数、词汇提取、里程碑检测）。

### 升级2：消息来源标识（Source Type）

**修改文件**：`dualsoul/database.py`（MIGRATIONS_V2），`dualsoul/routers/social.py`，`web/index.html`

#### 核心思想
双生命社交的信任基础：对话中的每个人始终清楚"这条消息是谁发的"——真人实时在打字，还是分身自动生成的。

#### source_type 字段
```
human_live          真人实时发送
twin_auto           分身自动生成（自主社交/自动回复/关系维护）
twin_draft_confirmed 分身起草，真人确认后发送
twin_standby        分身守候模式
```

#### 前端显示
分身自动发送的消息气泡上显示赛博风小标签 **✦ 分身**，不突兀但清晰。

### 升级3：关系双边许可（Twin Permission）

**修改文件**：`dualsoul/database.py`（MIGRATIONS_V2），`dualsoul/twin_engine/autonomous.py`，`dualsoul/routers/social.py`，`web/index.html`

#### 核心思想
接受好友请求 ≠ 接受对方分身进入你们的关系。分身代表主人主动联系某人之前，需要对方明确许可。

#### 流程
1. 两人成为好友，`twin_permission` 默认 `pending`
2. 分身第一次准备主动联系对方时，先检查权限
3. 如果是 `pending`：发系统通知「[用户名] 的分身想代表他和你保持联系，是否允许？」
4. 对方点"允许" → `granted`，分身正常运作
5. 对方点"拒绝" → `denied`，分身不再主动联系（真人聊天不受影响）

#### API
```
POST /api/social/friends/{friend_id}/twin-permission
body: {"permission": "granted"} 或 {"permission": "denied"}
```

### 升级4：分身状态机（7种状态）

**新增文件**：`dualsoul/twin_engine/twin_state.py`
**修改文件**：`dualsoul/routers/ws.py`，`web/index.html`

#### 7种状态
```
HUMAN_ACTIVE          🟢  真人主导，分身静默
TWIN_RECEPTIONIST     🔵✦ 分身接待中（真人离线超过阈值）
TWIN_DRAFT_PENDING    ⏳  分身起草，等真人确认
TWIN_STANDBY          💤  分身守候，不做判断
TWIN_MAINTENANCE      🌙  分身维护关系温度，禁止承诺
MEMORIAL              🕯️  纪念模式，仅供查阅
FROZEN                ❄️  冻结
```

#### 实时广播
- 用户上线 → WebSocket 广播 `twin_state_change: human_active` 给所有在线好友
- 用户下线 → 广播 `twin_state_change: twin_receptionist`（有分身设置）或 `twin_standby`

#### 前端
聊天窗口顶部，对方名字旁边实时显示状态图标，让用户始终知道"我现在在和谁说话"。

### 升级5：五阶段社会化成长路径

**修改文件**：`dualsoul/twin_engine/life.py`，`web/index.html`

#### 核心思想
原有的 XP/等级系统是游戏化数字，缺乏意义。重新映射为有叙事感的五阶段社会化成长路径，对应文档中"工具型→代理型→协作型→关系型→生命型"的智能体成熟模型。

#### 五阶段定义
```
阶段1：工具分身（LV.1-2）
  核心能力：自动回复、代接消息
  解锁条件：发送5条消息
  描述：你的分身刚刚醒来，能够接待来访的朋友。

阶段2：代理分身（LV.3-5）
  核心能力：主动联系、管理关系
  解锁条件：累计20条消息 + 1位好友
  描述：你的分身开始主动行动，能在你不在时维持关系。

阶段3：协作分身（LV.6-9）
  核心能力：情绪感知、识别场合
  解锁条件：累计50条消息 + 3位好友 + 连续活跃7天
  描述：你的分身开始读懂对方的情绪，知道什么时候说什么话。

阶段4：关系分身（LV.10-14）
  核心能力：维护关系体、积累共同记忆
  解锁条件：累计100条消息 + 5位好友 + 关系体温度均值≥60℃
  描述：你的分身开始真正参与关系的建设，不只是传话，而是共同积累记忆。

阶段5：生命分身（LV.15+）
  核心能力：持续人格、双生命社交完整参与
  解锁条件：累计500条消息 + 10位好友 + 至少1段关系体满30天连续
  描述：你的分身已具备持续的人格存在，能独立参与双生命社交世界。
```

#### 前端展示
生命面板中以"成长卡片"形式展示当前阶段名称、能力描述、下一阶段解锁进度，替代纯数字 XP 进度条。

## 数据库迁移说明

本版本数据库升级完全向后兼容，无需手动操作：

- `SCHEMA_V6` 使用 `CREATE TABLE IF NOT EXISTS`，旧数据库只会新增表
- `MIGRATIONS_V2` 使用 `ALTER TABLE ... ADD COLUMN`，已存在列会被 `OperationalError` 静默跳过
- 旧用户数据（消息、好友、分身设置）完全保留
- 现有好友关系的 `twin_permission` 默认值为 `pending`（等待首次分身联系时触发许可请求）
